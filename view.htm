2.1.1 Usability and UX

Access ‘Your saved links’ from the bottom of the page

Your saved links

View and edit all saved links

While our focus in this block is the usability of the web, we would like you to contextualise it within the overarching concept of user experience in general. In fact, usability and user experience (UX) are intrinsically related in software and web development. While they are distinct elements, they interact and influence each other significantly: UX encompasses the broader experiential and emotional aspects of the user’s interaction with a software, while usability focuses on the practical elements of interaction. Usability can therefore be seen as a crucial part of UX – to achieve a positive and successful user experience, it is essential to ensure high usability.

Activity 2.5 Industry perspective on web usability

Timing: 10 minutes

Spend five to ten minutes watching Video 2.3, where two developers from Isotoma explain what UX is, why it is important and how Isotoma approaches it. Write down a few notes in your learning journal about why UX is important.

Transcript: Video 2.3 User experience

ANDY THEYERS
My name is Andy and I’m one of the founders of Isotoma. I’m involved in the beginnings of projects, where we try and work out what our customers need us to build.
JAMES MITCHELL
Hi, I’m James. I’m a UX architect at Isotoma.
NARRATOR
Why is user experience important? And how does Isotoma approach it?
ANDY THEYERS
The term user experience describes the way a user interacts with a piece of software and how it’s been designed to meet their needs.
So when a product is frustrating to use or it’s hard to understand, that’s bad user experience. Whereas the best user experience is often invisible – we don’t even notice how easy something is to use. Sometimes user experience can be crucially important. On an ecommerce site, for example, bad user experience could cost thousands of pounds in lost transactions every day or even every hour.
Isotoma approaches user experience in radically different ways for different clients and projects. For example, if it’s a system which is used once every ten years, like renewing your passport, you need a very different user experience from a system which is used all day, every day, like, for example, your email client. We use user experience to establish guiding principles that feed throughout a project. So every feature is checked against each of these principles, making sure we’re building something that meets the needs of the user.
It’s also really important to understand that good user experience practice leads through to good accessibility and good SEO. So, done well, user experience is vitally important.
NARRATOR
What does a UX architect do at Isotoma?
JAMES MITCHELL
So a UX architect is involved at every point in the product’s life cycle. But mostly we’re involved in the initial stages, which is what we call discovery. That’s where we’re looking to identify the different requirements for the product, including its functionality, who the users might be, where it might be used and the other products it might be used alongside.
I’m doing a lot of this in conjunction with Andy, who is the internal product owner, in order to understand the users and the context of the product, in order to create a solution that’s not only going to fill the requirements for functionality, but it’s also going to enable and empower the target users, being intuitive and easy to integrate into their current workflows.
So once we understand the context of the product, we then do further research, hoping to gain insights that will enable us to create a better product. That might include things like usability testing on current or competitor products, interviews with users, surveys or any other way of gaining insights from users where we’re communicating with users.
From there, we would aim to define the new product using artefacts such as wireframes, information architecture, diagrams and user stories. We’d refine these alongside the developers and QA to produce a specification for the customer to sign off and for us to be able to show what we’re going to build.
After this, as UX, our cadence slows and we will be there to edit and answer questions as required through both development and testing.
When the product is launched, we’d still be involved. That might be iterating on the current interface, using insights from analytics. Or we might be doing further user testing, taking those solutions that we find and testing through A/B testing. Or, if we have much bigger requirements, we might be doing a full discovery process, which would involve research, wireframe and implementing those new features.
NARRATOR
What resources does a UX architect produce at Isotoma?
JAMES MITCHELL
So the outputs we create really depend on the client and what activities we’ve done during the discovery process. But in addition to wireframes, we would often deliver things like a style guide or a design system, which would define the look and feel of a product, including the typography, the colours and the components.
Style guides are usually only for a single product, whereas a design system might define it across multiple products, devices and even formats.
We might also produce affinity maps, which are ways that we gather and group the different insights that we got from research, whether that was interviews or surveys or any other research that we’ve done.
We might also produce an information architecture diagram, which is where we’ve defined how we’d structure the product for the user, basically. The developers might also use this as a basis for their data model design.
Another thing we might create is a user role definition table, which is where we define the different roles that would exist within the product. Again, the developers might use this during implementation to define the different user roles.
At Isotoma, we use Sketch to create most of our wireframes. We’ve also used Figma, XD and Balsamiq in the past. Figma is usually used because the client requests it for collaboration. We use Sketch because it’s offline. Figma is mostly online. Balsamiq we’ve used in the past just because it has some features to transition between low- and high-fidelity wireframes if we want to use those for testing.
NARRATOR
Where can user testing and user monitoring add most value to a project?
ANDY THEYERS
User testing is expensive and takes a lot of time, so it must be used selectively. It usually happens at a stage where the most is unknown, so at the start of the project or when we’re considering a major change to existing software.
To carry out user testing, we get together example users and we run them through either examples of the software itself, so working software, or we take them through paper prototypes, where we use the wireframes to see if they can carry out task completion.
After we’ve gone live, we can use real user monitoring, or RUM, instead. This is where we track actual usage of the software. So, for example, we see users’ mouse movements, we can see where they click, we can see the buttons they press and we can see the text that they enter into boxes. So, for example, we can see how they’re using the search feature and whether or not they really do use the big button that you provided and are considering making changes to.
RUM is an extremely powerful tool, which can be set to capture any percentage, right up to a hundred per cent, of user interactions. And some systems can even provide video simulations of the user interactions which have been captured.
Combined, these two techniques provide a wealth of qualitative and quantitative data to inform decisions about requirements and priorities that really reflect our users’ needs.

Comment

Some thoughts to consider:

  • When a product is not easy to use or hard to understand, that’s bad user experience.
  • Bad user experience can cost companies money.
  • User experience should be approached differently for different clients and projects.
  • User experience is used to establish guidelines to support the development of the project.
